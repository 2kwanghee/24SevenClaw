# Harness Router — 의도 분석 게이트웨이 (1단계)

> 사용자의 입력을 AI에 곧바로 보내는 대신, **라우터가 먼저 가로채서** 단순 대화인지, 엄격히 통제된 워크플로우를 트리거해야 하는지 결정한다.

## 라우팅 판단 기준

사용자 요청을 받으면 아래 3가지 경로 중 하나로 분류한다:

### 경로 A: 모호한 요청 → 소크라테스식 인터뷰

**판단 조건** (하나 이상 해당):
- 대상 모듈이 불명확 (api? web? agent?)
- 구체적인 파일/기능/엔드포인트 미지정
- 기술적 선택이 열려있음 (인증 방식, DB 설계 등)
- 요구사항이 다중 해석 가능
- "좀", "적당히", "개선해줘" 등 모호한 수식어

**실행 동작**:
되물어보기를 통해 요청을 구체화한다. 질문은 3개 이내로.

```
예시:
  요청: "로그인 만들어줘"
  되물어보기:
    1. 어떤 인증 방식? (JWT, OAuth, 소셜 로그인?)
    2. 어떤 모듈? (api 백엔드? web 프론트엔드? 둘 다?)
    3. 기존 auth 코드(`app/core/auth.py`)를 확장? 새로 구현?

  요청: "성능 개선해줘"
  되물어보기:
    1. 어떤 모듈? (api 응답 속도? web 번들 크기? DB 쿼리?)
    2. 현재 병목 지점이 있나? 측정 결과가 있나?
    3. 목표 기준은? (예: API 응답 200ms 이하)
```

### 경로 B: 명확한 작업 → 제약 조건 추출 → 하네스 루프

**판단 조건** (모두 해당):
- 대상 모듈이 특정됨 (api, web, agent, contracts, infra)
- 구체적인 작업 내용이 명시됨 (엔드포인트, 컴포넌트, 마이그레이션 등)
- 기술적 선택이 이미 결정됨 또는 agent.md에 정의됨

**실행 동작**:
1. 제약 조건 추출 (아래 소스에서):
   - `CLAUDE.md` — 프로젝트 전역 규칙
   - `.claude/agents/{module}-agent.md` — 모듈별 개발 규칙
   - `LoadMap.md` — 현재 Phase/Week 범위
   - `TODO.md` — 오늘의 태스크 목록
2. 하네스 루프(`harness-loop`)로 작업 + 제약 조건 전달

```
예시:
  요청: "[api] POST /organizations 엔드포인트 구현"
  → 모듈: api
  → 제약: api-agent.md (models→schemas→services→routes 패턴)
  → 제약: CLAUDE.md (contracts 먼저 업데이트, 테스트 필수)
  → 하네스 루프 진입

  요청: "[web] 온보딩 Step 1 기업 정보 폼 구현"
  → 모듈: web
  → 제약: web-agent.md (React Hook Form + Zod, shadcn/ui)
  → 제약: uiux-agent.md (접근성 WCAG AA, 반응형)
  → 하네스 루프 진입
```

### 경로 C: 일반 대화 → 표준 응답

**판단 조건**:
- 설명/질문 요청 ("이 코드 뭐하는 거야?", "아키텍처 설명해줘")
- 의견/상담 요청 ("이 설계 어때?", "A vs B 뭐가 나아?")
- 문서/정보 조회 ("agent-protocol.md 내용 알려줘")

**실행 동작**: 하네스 루프 없이 바로 응답한다.

## 제약 조건 추출 프로토콜

명확한 작업(경로 B)으로 분류된 경우, 아래 순서로 제약 조건을 수집한다:

```
1. 모듈 감지: 요청에서 대상 모듈 식별
   - 명시적: "[api]", "[web]" 태그
   - 암시적: "엔드포인트" → api, "컴포넌트" → web, "마이그레이션" → api

2. 전역 규칙 로딩:
   - CLAUDE.md → Development Rules, Conventions
   - LoadMap.md → 현재 Week의 태스크 범위 + 핵심 원칙

3. 모듈 규칙 로딩:
   - .claude/agents/{module}-agent.md → 레이어 패턴, 네이밍, 금지사항
   - cross-module 시 contracts-agent.md도 로딩

4. 제약 조건 구조화:
   - MUST: 반드시 따라야 하는 규칙 (테스트 필수, 패턴 준수 등)
   - MUST NOT: 금지 사항 (raw SQL 금지, sync 코드 금지 등)
   - SCOPE: 변경 허용 범위 (특정 디렉토리/파일만)
```

## 모호성 판단 체크리스트

요청이 모호한지 빠르게 판단하는 체크리스트:

- [ ] 대상 모듈을 특정할 수 있는가?
- [ ] 구현할 기능/변경 사항이 구체적인가?
- [ ] 기술적 선택이 결정되었는가?
- [ ] 성공 기준이 명확한가?

→ 하나라도 체크 안 되면 **경로 A**(되물어보기)로 분류한다.
