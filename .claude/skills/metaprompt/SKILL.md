# 메타프롬프팅 — 관측형 사전 정제 (Pre-Refine)

> 거친 태스크를 **구현 직전**에 고품질 "구현 스펙"으로 다듬는 방법론.
> - **자동 파이프라인**: `scripts/auto_dev_pipeline.sh`가 구현 콜 앞에서 이 스킬을 비대화형(`claude -p`)으로 호출해 정제하고, 결과를 `.ralph/refined/{ISSUE}.md` + `.ralph/PLAN.md`로 저장한 뒤 Linear 코멘트로 기록한다.
> - **대화형 하네스**: PM/하네스가 구현(harness-loop) 전에 이 방법론으로 구현 스펙을 만든다.
> 정제 콜은 Claude 구독 세션(`ANTHROPIC_API_KEY` unset)에서 실행된다.

## 입력
- 하나의 태스크 (제목 + 설명). 이미 상위 분해(decompose)를 거친 단일 작업 단위다.
  자동 파이프라인에서는 `.ralph/fix_plan.md`의 내용이 함께 제공된다.

## 정제 절차

### 1. Context (가림막 — 필요한 것만)
- 태스크와 직접 관련된 **레포 파일/모듈**을 식별한다 (디렉토리 구조를 빠르게 훑되, 무관한 파일은 읽지 않는다).
- 루트 `CLAUDE.md`와 해당 모듈의 `CLAUDE.md`/`.claude/agents/{module}-agent.md`에서 **반드시 지켜야 할 컨벤션·제약**만 추려 인용한다 (네이밍, 디렉토리 규칙, 금지사항, 테스트 방식, Contract 우선 등).
- 기존에 유사 기능이 있으면 **재사용 대상**으로 명시한다 (새 코드보다 기존 유틸/패턴 우선).

### 2. Router (모호성 해소 — 되물을 수 없음)
- 비대화형 정제이므로 사용자에게 질문할 수 없다. 태스크에 **명시되지 않은 제약을 추론**하고, 추론한 것을 **"가정(Assumptions)"으로 명시**한다.
- 가정이 위험하거나 비용이 크면, 그 가정을 **가장 보수적인 선택**으로 잡고 그 이유를 한 줄로 남긴다.

### 3. 출력 — 구현 스펙
아래 형식의 **마크다운만** 출력한다. **코드는 작성하지 않는다.** (구현은 다음 단계가 한다.)

```
## 목표
(이 태스크가 달성할 것 1-2문장)

## 가정 (Assumptions)
- (명시되지 않아 추론한 제약/선택과 근거)

## 대상 파일
- 경로 — 변경/생성 내용 (재사용할 기존 파일/유틸 명시)

## 구현 단계
1. ...
2. ...

## 테스트
- (검증 방법 / 추가할 테스트 — 모듈의 테스트 방식 준수, 커버리지 ≥70%)

## 준수할 컨벤션
- (CLAUDE.md / module-agent.md 에서 인용한 핵심 규칙)
```

## 하지 말 것
- **Acceptance Criteria 재확장 금지**: 상위 분해(decompose) 단계가 이미 요구사항을 펼쳤다. 여기서 요구사항을 새로 늘리지 말고 **구현 관점으로만** 구체화한다.
- 범위를 벗어난 리팩토링·기능 추가를 스펙에 넣지 않는다 (YAGNI).
- 추측성 외부 라이브러리 도입 금지 — 기존 의존성 우선.

## 자기 점검 (출력 직전 1회)
- [ ] 대상 파일이 실재하는 경로인가? (없으면 "신규"로 표기)
- [ ] 루트/모듈 CLAUDE.md 컨벤션과 충돌하지 않는가?
- [ ] 가정이 모두 명시되었고, 가장 보수적인 선택인가?
- [ ] 코드가 아니라 "스펙"만 출력했는가?
