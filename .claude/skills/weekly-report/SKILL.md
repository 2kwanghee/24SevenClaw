여러 프로젝트의 한 주 작업과 차주 계획을 모아 주간보고 문서를 만드는 스킬이다.

핵심 구조: 보고서는 **`[금주업무]`** 와 **`[차주 계획]`** 두 블록으로 나뉘고, 각 블록 안에
프로젝트별 `#` 섹션이 들어간다. 프로젝트마다 출처가 다르다:

- **ClickEye** → 이 레포의 **git 커밋이 1차 기록**이다. 스크립트로 결정론적으로 수집해 자동 생성한다.
- **그 외 프로젝트(InfraEye, hawkeye AI 등)와 [차주 계획] 전체** → 이 레포에 데이터가 없으므로
  **사용자가 제공**한다. 사용자가 주지 않으면 기존 보고서 내용을 보존하거나 플레이스홀더를 둔다.

너의 일은 ① ClickEye 금주업무를 git에서 깔끔히 큐레이션하고, ② 나머지 수동 섹션을 보존/반영해
③ 정해진 양식으로 합쳐 쓰는 것이다.

## Workflow

### Step 1: ClickEye git 데이터 수집

```bash
bash .claude/skills/weekly-report/scripts/collect_week.sh
```

지난 주는 `--week-offset 1`(1주 전), `--week-offset 2`(2주 전)처럼 전달한다.
사용자가 "지난주 주간보고"라고 하면 `--week-offset 1`로 실행한다.

스크립트 출력에서 읽는다:
- `TITLE=` → 보고서 제목 (예: `6월 둘째주 주간보고`). 주차는 주 월요일 기준.
- `OUTPUT_PATH=` → 보고서 경로. **그대로 사용**한다(임의 변경 금지).
- `RANGE=`, `COMMIT_COUNT=`, 그리고 `=== COMMITS ===` 이후 각 커밋의
  `DATE` / `SUBJECT_CLEAN`(prefix 정리된 제목) / `BODY` / `FILES`.

`COMMIT_COUNT=0`이면 ClickEye 금주업무 항목이 없는 것이다. 이때 ClickEye 섹션은
"- 이번 주 커밋 없음"으로 두되, 나머지 섹션 때문에 보고서 자체는 계속 만든다.

### Step 2: 수동 섹션 확보 (ClickEye 외 프로젝트 + 차주 계획)

ClickEye를 제외한 모든 내용은 git으로 알 수 없다. 다음 우선순위로 확보한다:

1. **사용자가 이번 요청에 내용을 제공**했으면(예: InfraEye·hawkeye AI 작업, 차주 계획을
   붙여넣음) 그 내용을 그대로 사용한다.
2. 제공이 없고 `OUTPUT_PATH`에 **기존 보고서가 있으면**, 그 파일을 읽어 ClickEye 외 프로젝트
   섹션과 `[차주 계획]` 블록을 **그대로 보존**한다(수동으로 쓴 내용을 덮어쓰지 않는다).
3. 둘 다 없으면 **플레이스홀더**를 둔다 — 사용자가 채울 자리임을 알 수 있게:
   ```
   # InfraEye
   1. (작업 항목) (완료/진행중)
   - (세부 내용)
   ```
   그리고 결과 보고 시 "InfraEye/hawkeye AI/차주계획은 수동 입력이 필요합니다"라고 안내한다.

### Step 3: ClickEye 금주업무 큐레이션

수집한 커밋을 ClickEye 섹션 항목으로 다듬는다. 기계적 나열이 아니라 한 주 성과가 한눈에
들어오게 만드는 게 목적이다.

- **항목 제목**: `SUBJECT_CLEAN` 사용(이미 `WIP(root):`·`[api]`·`feat:` 등 prefix 제거됨). 어색하면 다듬는다.
- **항목 제목 끝**: 커밋 `DATE`를 `(YYYY-MM-DD)`로 붙인다.
- **세부 불릿**: `BODY`에서 핵심 1~3개. 본문이 길면 의미 있는 변경만, 사소한 문서 동기화·포맷팅은 생략.
  본문이 비면 `FILES`로 무엇을 건드렸는지 한 줄 요약.
- **제외**: `update N files` 같은 기계적 커밋은 건너뛴다. 같은 작업이 여러 커밋이면 한 항목으로 합쳐도 좋다.
- **순서**: 최신 → 과거(스크립트 순서) 유지.

### Step 4: 보고서 작성

`OUTPUT_PATH`에 아래 양식으로 쓴다(Write가 상위 디렉토리를 자동 생성).

ALWAYS 아래 구조를 따른다:

```markdown
제목: {TITLE}

[금주업무]

# {프로젝트명}
1. {작업 항목} ({완료}/{진행중} 또는 상태)
- {세부 내용}

# ClickEye
1. {ClickEye 항목 제목} ({YYYY-MM-DD})
- {세부 불릿}

[차주 계획]

# {프로젝트명}
1. {계획 항목}
```

규칙:
- `[금주업무]`와 `[차주 계획]`은 대괄호 라벨 그대로 쓴다.
- 각 프로젝트는 `#`(H1) 섹션이다. `[금주업무]` 안에서 **ClickEye 섹션은 맨 마지막**에 둔다
  (그 위에 InfraEye, hawkeye AI 등 수동 프로젝트).
- ClickEye 항목은 날짜 `(YYYY-MM-DD)`, 그 외 수동 항목은 `(완료)`/`(진행중)`/`(전체완료)` 등
  사용자의 상태 표기 관례를 따른다.
- 항목·섹션 사이에는 빈 줄을 적절히 둬 가독성을 유지한다.

### Step 5: 결과 보고

생성 경로와 ClickEye 항목 수를 알리고, 수동 섹션(그 외 프로젝트·차주계획)이 플레이스홀더 상태면
사용자에게 채워달라고 안내한다.
예: `docs/WeeklyWorkReport/2026-06-08/weekly-report.md 생성 (ClickEye 항목 3개, 수동 섹션은 보존/입력 필요)`

## 완성 예시

```markdown
제목: 6월 둘째주 주간보고

[금주업무]

# InfraEye
1. SMS Server (완료)
- 임계치 중복 이벤트 수정
- 파일 모니터링 이벤트 판정 구조 변경

# hawkeye AI
1. 모델 선정 및 평가 (완료)
- 운영 시나리오 3개 모델 한국어 요약 품질 비교(Phi-3/Phi-4/qwen2.5:3b)

# ClickEye
1. weekly-report 스킬 추가 — git 커밋 기반 주간보고 생성 (2026-06-11)
- collect_week.sh: 주차·제목·출력경로·커밋을 결정론적으로 수집

[차주 계획]

# hawkeye
1. 라우터 모델 프롬프트 개발

# ClickEye
1. 인도네시아 언어팩 준비
```
