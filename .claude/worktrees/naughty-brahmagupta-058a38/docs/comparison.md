# ClickEye - Architecture Comparison

## 1. 개요

ClickEye의 "클라우드 컨트롤 플레인 + 고객 서버 실행 플레인" 아키텍처를 유사 모델들과 비교 분석한다.

---

## 2. 비교 대상

| 플랫폼 | 핵심 모델 | 출시 |
|--------|----------|------|
| **GitHub Actions (Self-hosted Runner)** | CI/CD 워크플로 실행을 사용자 서버에 위임 | 2019 |
| **Ansible Tower / AWX** | IT 자동화 플레이북을 원격 서버에서 실행 | 2012/2017 |
| **GitLab Runner** | CI/CD 파이프라인을 사용자 서버에서 실행 | 2015 |
| **ClickEye** | AI 에이전트 개발 환경 + 프로세스를 고객 서버에서 실행 | 2026 |

---

## 3. GitHub Actions Self-hosted Runner

### 3.1 아키텍처

```
[GitHub.com]                    [고객 서버]
┌──────────────┐               ┌──────────────┐
│ GitHub       │◄─── HTTPS ───│ Self-hosted  │
│ Actions      │  (Runner →   │ Runner       │
│ Service      │   GitHub)    │              │
│              │               │ - 워크플로    │
│ - 워크플로    │               │   실행       │
│   정의       │               │ - 빌드       │
│ - 트리거     │               │ - 테스트     │
│ - 로그 표시  │               │ - 배포       │
└──────────────┘               └──────────────┘
```

### 3.2 동작 방식
1. Runner가 GitHub에 **아웃바운드** HTTPS 롱폴링으로 연결
2. 새 워크플로가 트리거되면 GitHub가 Runner에 Job 할당
3. Runner가 로컬에서 Job 실행
4. 실행 로그를 GitHub에 실시간 스트리밍
5. 결과(성공/실패)를 GitHub에 보고

### 3.3 ClickEye와의 유사점
| 유사점 | 설명 |
|--------|------|
| 연결 방향 | Runner/Agent 모두 클라우드로 **아웃바운드** 연결 |
| 실행 위치 | 실제 작업은 고객 서버에서 수행 |
| 명령 수신 | 클라우드에서 작업 정의 → 로컬에서 실행 |
| 상태 보고 | 실행 결과를 클라우드로 보고 |

### 3.4 차이점
| 항목 | GitHub Actions Runner | ClickEye Agent |
|------|----------------------|-------------------|
| **용도** | CI/CD 파이프라인 실행 | AI 에이전트 개발 환경 전체 관리 |
| **상태** | Stateless (Job 단위) | Stateful (환경 지속 유지) |
| **통신** | HTTPS 롱폴링 | WebSocket (실시간 양방향) |
| **데이터** | 빌드 로그 GitHub 전송 | 코드/데이터 고객 서버에만 보관 |
| **범위** | 빌드/테스트/배포 | 환경 구성 + 개발 + 빌드 + 실행 |
| **AI 연동** | 없음 | Claude 인스턴스 관리 + 티켓 전달 |

### 3.5 우리가 배울 점
- **Runner 등록 흐름**: 토큰 기반 등록은 검증된 패턴 → 그대로 채택
- **자동 업데이트**: Runner가 자체 업데이트 가능 → Agent에도 적용
- **Runner Group**: 여러 Runner를 그룹으로 관리 → 여러 Agent 관리에 참고
- **레이블 시스템**: Runner에 레이블 → Agent에 capabilities 태그

## 4. Ansible Tower / AWX

### 4.1 아키텍처

```
[Ansible Tower/AWX]             [관리 대상 서버들]
┌──────────────┐               ┌──────────────┐
│ Tower/AWX    │── SSH/WinRM ──│ 서버 A       │
│ (웹 UI)     │               └──────────────┘
│              │               ┌──────────────┐
│ - 인벤토리   │── SSH/WinRM ──│ 서버 B       │
│ - 플레이북   │               └──────────────┘
│ - 잡 스케줄  │               ┌──────────────┐
│ - RBAC      │── SSH/WinRM ──│ 서버 C       │
│ - 감사 로그  │               └──────────────┘
└──────────────┘
```

### 4.2 동작 방식
1. 관리자가 Tower UI에서 Playbook + Inventory 구성
2. Job 실행 → Tower가 대상 서버에 SSH 접속
3. Ansible 모듈을 대상 서버에서 실행
4. 실행 결과를 Tower DB에 저장

### 4.3 ClickEye와의 유사점
| 유사점 | 설명 |
|--------|------|
| 중앙 관리 | 웹 UI에서 원격 서버 관리 |
| 인벤토리 | 관리 대상 서버 목록 관리 |
| 템플릿 | 작업 템플릿(Playbook) 정의 |
| RBAC | 역할 기반 접근 제어 |
| 감사 로그 | 모든 작업 이력 추적 |

### 4.4 차이점
| 항목 | Ansible Tower/AWX | ClickEye Agent |
|------|-------------------|-------------------|
| **연결 방향** | Tower → 서버 (SSH, 인바운드) | Agent → Cloud (WebSocket, 아웃바운드) |
| **에이전트** | Agentless (SSH만) | Agent 데몬 필수 설치 |
| **용도** | 인프라 자동화 (구성 관리) | AI 개발 환경 + 프로세스 오케스트레이션 |
| **상태** | 멱등성 기반 (선언적) | 이벤트 기반 (명령적) |
| **방화벽** | SSH 인바운드 필요 | 아웃바운드만 (방화벽 친화적) |
| **AI 연동** | 없음 | Claude + 에이전트/스킬/MCP |

### 4.5 우리가 배울 점
- **Credential 관리**: Tower의 Credential Vault → Agent의 API 키 관리에 참고
- **Job Template**: 재사용 가능한 작업 템플릿 → 환경 템플릿 시스템에 적용
- **Survey (사전 입력)**: Job 실행 전 사용자 입력 → 티켓 발행 시 파라미터 입력
- **알림 통합**: Slack/Email 알림 → 우리도 작업 완료 알림에 적용
- **RBAC**: 역할 기반 권한 → 팀 기능 추가 시 참고

## 5. GitLab Runner

### 5.1 아키텍처

```
[GitLab.com / Self-hosted]      [고객 서버]
┌──────────────┐               ┌──────────────┐
│ GitLab       │◄─── HTTPS ───│ GitLab       │
│ CI/CD        │  (Runner →   │ Runner       │
│ Service      │   GitLab)    │              │
│              │               │ - Executor   │
│ - .gitlab-ci │               │   (Docker,   │
│ - 파이프라인  │               │    Shell,    │
│ - 결과 표시  │               │    K8s)      │
└──────────────┘               └──────────────┘
```

### 5.2 ClickEye에 참고할 점
- **Executor 개념**: Docker, Shell, Kubernetes 등 다양한 실행 방식 → Agent의 handler 패턴에 적용
- **캐시/아티팩트**: Runner 간 캐시 공유 → 프로젝트 간 이미지 캐시에 적용
- **태그 매칭**: Runner 태그로 특정 Job 할당 → Agent 능력 기반 작업 라우팅에 적용

## 7. ClickEye의 차별점 요약

### 7.1 기존 모델 대비 독자적 특성

1. **AI 에이전트 네이티브**: CI/CD나 인프라 자동화가 아닌, AI 에이전트 기반 개발 프로세스 전체를 관리
2. **완전한 데이터 주권**: 로그마저도 요약만 클라우드에 전송. 기존 플랫폼들은 로그/아티팩트를 클라우드에 저장
3. **Stateful Agent**: Job 단위가 아닌, 지속적인 개발 환경을 유지. 환경은 Agent가 관리
4. **실시간 양방향**: WebSocket으로 티켓 전달 → 작업 진행 → 실시간 모니터링까지 끊김 없는 UX
5. **레지스트리 통합**: 에이전트/스킬/MCP를 클라우드 마켓플레이스에서 선택 → 고객 서버에 자동 설치

### 7.2 기술적 우위

```
GitHub Actions:  "코드 푸시 → 빌드/테스트/배포 자동화"
Ansible Tower:   "인프라 구성을 코드로 관리"
ClickEye:     "AI 에이전트에게 개발 작업을 지시하고, 전체 개발 라이프사이클을 오케스트레이션"
```

## 8. 아키텍처 결정 근거

### Q: 왜 Agentless(Ansible 방식)가 아닌 Agent 방식인가?
- SSH 인바운드 포트 개방은 보안 리스크
- 지속적인 환경 상태 관리가 필요 (Stateful)
- 실시간 양방향 통신이 필요 (티켓 전달 + 모니터링)
- Docker 관리, Claude 프로세스 관리 등 로컬 데몬이 필수적인 작업

### Q: 왜 HTTPS 롱폴링이 아닌 WebSocket인가?
- 실시간 로그 스트리밍, 작업 진행 상황 보고에 롱폴링은 비효율
- 양방향 통신이 빈번 (명령 전달 + 상태 보고)
- 연결 유지 비용 대비 실시간성 이점이 큼

### Q: 왜 데이터를 클라우드에 저장하지 않는가?
- 고객사의 데이터 주권 보장 (규정 준수: GDPR, 국내법 등)
- 클라우드 인프라 비용 절감 (저장소, 컴퓨팅)
- 보안 사고 시 영향 범위 최소화 (클라우드 해킹 → 고객 코드 유출 없음)
- 고객이 원하면 완전 오프라인으로 전환 가능 (라이센스 검증만 필요)
