# 24SevenClaw - Architecture Overview

## 1. 시스템 개요

24SevenClaw는 **컨트롤 플레인 / 실행 플레인** 분리 아키텍처를 채택한 AI 에이전트 개발 오케스트레이션 플랫폼이다.

- **컨트롤 플레인 (Cloud)**: 개발 프로세스 관리, 라이센스, 레지스트리, UI
- **실행 플레인 (Customer Server)**: 실제 코드 실행, Docker 환경, 데이터 저장

이 모델은 GitHub Actions의 Self-hosted Runner, Ansible Tower/AWX, GitLab Runner와 동일한 패턴을 따른다.

## 3. 컨트롤 플레인 (Cloud) 상세

### 3.1 역할과 책임

| 역할 | 설명 |
|------|------|
| **사용자 인증** | 계정 관리, 로그인, JWT 토큰 |
| **라이센스 관리** | 프로젝트 단위 라이센스 발급/검증 |
| **레지스트리** | 에이전트/스킬/MCP 카탈로그 (회사 IP) |
| **프로젝트 관리** | 프로젝트 메타데이터, 설정 참조 |
| **티켓/이슈** | 개발 작업 발행, 트래킹 |
| **오케스트레이션** | Agent에게 명령 전달, 상태 수집 |
| **모니터링** | Agent 연결 상태, 작업 진행 현황 |

### 3.2 저장하는 데이터

```yaml
저장함:
  - 사용자 계정 (email, password_hash)
  - 라이센스 정보 (프로젝트별)
  - 프로젝트 메타데이터 (이름, 설명, 설정 참조)
  - 에이전트/스킬/MCP 레지스트리 (우리 IP)
  - 프로젝트별 에이전트/스킬/MCP 설정 매핑
  - Agent 연결 정보 (ID, 마지막 접속, 상태)
  - 티켓/이슈 (제목, 설명, 상태, 우선순위)
  - 티켓 이벤트 로그 (상태 변경 기록)

저장하지 않음:
  - 고객 소스 코드
  - 빌드 결과물/아티팩트
  - 실행 로그 원본 (요약만 수신)
  - Git 저장소 데이터
  - 고객 환경 설정 파일 원본
  - 고객의 비즈니스 데이터
```

## 5. 통신 아키텍처

### 5.1 연결 방향

```
Agent ────(outbound)────► Cloud WebSocket Hub

왜 이 방향인가?
- 고객 서버는 대부분 방화벽 뒤에 있음
- 인바운드 포트 오픈 불필요
- Agent가 클라우드에 아웃바운드 연결 → 방화벽 통과
- HTTPS/WSS(443) 포트만 사용
```

### 5.2 통신 프로토콜 요약

```
Agent → Cloud:
  - agent.register    # 최초 등록
  - agent.heartbeat   # 주기적 상태 보고 (30초)
  - agent.status      # 환경/작업 상태 업데이트
  - agent.log         # 로그 스트리밍 (요약)
  - agent.result      # 작업 완료 결과

Cloud → Agent:
  - command.setup_env       # 환경 프로비저닝 명령
  - command.deploy_ticket   # 티켓 전달
  - command.build           # 빌드 실행
  - command.run             # 서비스 실행
  - command.stop            # 서비스 중지
  - command.destroy_env     # 환경 삭제
  - config.update           # 설정 변경
```

상세 프로토콜: `docs/agent-protocol.md` 참조

## 6. 보안 고려사항

### 6.1 통신 보안
- **TLS**: 모든 WebSocket 통신은 WSS(TLS) 사용
- **Agent 인증**: 등록 시 라이센스 키 + Agent 고유 토큰 발급
- **메시지 서명**: 각 명령에 HMAC 서명 (위변조 방지)

### 6.2 데이터 보안
- 고객 코드/데이터는 절대 클라우드에 전송하지 않음
- 클라우드에는 작업 결과 요약만 전달 (제목, 상태, 변경 파일 목록 정도)
- Agent가 보고하는 로그는 민감 정보 필터링

### 6.3 라이센스 보안
- 라이센스 키는 Agent 설정에 저장
- 주기적으로 클라우드와 라이센스 유효성 검증
- 만료/비활성 라이센스 → Agent 기능 제한 (읽기 전용 모드)

## 7. 확장성 설계

### 7.1 수평 확장
- **Cloud**: FastAPI + Redis로 WebSocket Hub 수평 확장 가능
- **Agent**: 고객 서버 1대당 1 Agent (N개 프로젝트 관리)

### 7.2 고가용성
- Cloud: 로드밸런서 + 다중 인스턴스
- Agent: 재연결 전략 (지수 백오프 + 최대 5분)
- Redis Pub/Sub: 다중 API 인스턴스 간 WebSocket 메시지 브로커링
